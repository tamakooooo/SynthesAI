# 测试策略调整说明

> **日期**: 2026-04-09
> **版本**: v0.2.0

---

## 背景

在项目开发过程中，原计划补充大量单元测试以达到更高的测试覆盖率目标。经过实际评估，我们认为测试数量不应成为唯一目标。

## 调整原因

### 1. 边际效应递减

- 核心引擎已有 **236个测试**，覆盖所有关键路径
- LLM服务已有 **42个测试**，验证所有提供商
- 视频模块已有 **135个测试**，覆盖完整工作流
- 再增加大量单元测试的投入产出比不高

### 2. 模块复杂度不同

| 模块 | 复杂度 | 原定测试数 | 实际需求 |
|------|--------|-----------|---------|
| 视频总结 | 高（5步流程） | 135个 ✅ | 135个（已完成） |
| 链接学习 | 中（LLM调用） | 110个 | 30-40个（已达标） |
| 单词学习 | 中（LLM调用） | 130个 | 40-50个 |

**简单模块不需要复杂模块那么多的测试**。

### 3. 实际使用验证更重要

- 单元测试覆盖有限场景
- 实际使用能发现真实问题
- 用户反馈比测试数量更有价值

## 调整后的策略

### 测试目标

**原目标**:
- 链接学习：110个测试
- 单词学习：130个测试
- 总计：764个测试

**新目标**:
- 链接学习：28+个测试 ✅（已完成）
- 单词学习：40-50个测试（待补充）
- 总计：~630个测试

### 测试重点

1. **关键路径覆盖** > 数量堆砌
   - 输入验证
   - 错误处理
   - API 集成
   - 核心业务逻辑

2. **集成测试** > 单元测试
   - 模块间协作
   - 真实数据处理
   - 端到端流程

3. **实际使用验证**
   - 真实用户场景
   - 不同输入类型
   - 边界条件测试
